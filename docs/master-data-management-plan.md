# 分類與項目管理功能規劃

更新日期：`2026-05-13`

## 1. 背景

目前分類與項目分成兩頁管理：

- `分類管理`
- `項目管理`

當資料量變多時，會出現幾個問題：

- 分類與項目之間需要來回切換，不容易建立對應關係
- 項目列表會變很長，難以快速找到同一分類底下的資料
- 不容易看出哪些分類是空的、哪些分類底下項目過多
- 維護排序、啟用狀態、幣別時，缺少群組脈絡

## 2. 目標

將分類與項目整合成同一個管理體驗，並以「分類群組」作為主要視角。

希望達成：

- 分類與項目的關係一眼可見
- 可以快速定位某個分類底下的所有項目
- 分類與項目都能在同一頁完成新增、編輯、啟停用
- 當資料量變多時，仍能靠搜尋、篩選與群組維持可管理性

## 3. 建議操作模式

建議改為單一 `Master Data` 管理頁，採用「左側分類、右側項目」或「分類群組展開」的整合式模式。

我推薦的主方案是：

- 左欄：分類列表
- 右欄：目前選中分類的項目列表

原因：

- 比單純長列表更容易建立層級
- 比全部展開的群組列表更容易在資料量大時維持可讀性
- 後續擴充搜尋、篩選、批次操作都比較自然

## 4. 資訊架構

### 4.1 頁面結構

同一頁分成三區：

1. 上方工具列
2. 左側分類清單
3. 右側項目區

### 4.2 上方工具列

提供全域操作：

- 搜尋分類 / 項目名稱 / 機構
- 篩選 `資產 / 負債`
- 篩選 `啟用 / 停用 / 全部`
- 切換排序方式
- `新增分類`
- `新增項目`

### 4.3 左側分類清單

每個分類顯示：

- 分類名稱
- 類型：資產 / 負債
- 項目數量
- 啟用狀態
- 排序

支援：

- 點擊切換當前分類
- 直接編輯分類名稱 / 類型 / 排序 / 啟用
- 快速新增該分類下的新項目

### 4.4 右側項目區

當左側選定分類後，右側顯示該分類底下的項目列表。

每個項目顯示：

- 項目名稱
- 機構
- 幣別
- 排序
- 啟用狀態
- 備註

支援：

- 在當前分類下新增項目
- 直接編輯項目
- 切換啟用狀態
- 調整排序

## 5. 建議互動流程

### 5.1 新增分類

1. 點 `新增分類`
2. 輸入名稱、類型、排序
3. 建立後自動選中該分類
4. 右側直接可以開始新增該分類的項目

### 5.2 新增項目

有兩種入口：

- 全域 `新增項目`
- 分類內 `新增此分類項目`

若是在分類內新增：

- 預設帶入當前分類
- 減少多餘選擇

### 5.3 維護分類與項目

日常最常見操作應該是：

1. 左側找到分類
2. 右側看該分類有哪些項目
3. 直接修改或新增

這會比現在分類頁 / 項目頁分開更符合實際使用流程。

## 6. MVP 範圍

第一階段建議先做以下內容：

- 新增一頁整合式管理頁
- 左側分類列表
- 右側依當前分類顯示項目
- 分類項目數量顯示
- 搜尋：名稱 / 機構
- 篩選：資產負債 / 啟用狀態
- 在分類內直接新增項目
- 分類與項目的 inline edit

這一版先不急著做：

- 拖曳排序
- 批次多選操作
- 多層群組
- 折疊全部分類的大型 tree view

## 7. 第二階段可再做

當資料更多後，可以再補：

- 拖曳排序
- 批次啟用 / 停用
- 批次移動分類
- 顯示「最近被哪些 snapshot 使用」
- 顯示「這個分類最後一次出現在哪個月結」
- 顯示「停用但歷史仍在使用」提示

## 8. API / 資料層調整建議

目前 API 已足夠支援基本 CRUD，但若要做整合頁，建議補一個整合查詢資料接口：

- `GET /api/master-data`

回傳內容可包含：

- categories
- items
- 每個分類的 itemCount

這樣可以減少前端組裝成本，也讓頁面初始化更乾淨。

另外建議補：

- 分類下項目數量統計
- 是否允許刪除分類的規則
- 分類停用後，是否同步影響項目顯示的規則

## 9. 刪除與停用策略

建議原則：

- `分類` 先不提供實體刪除，只提供停用
- `項目` 也優先停用而不是刪除

原因：

- 歷史 snapshot 仍可能參考這些主檔
- 停用比刪除安全很多
- 可以避免主檔整理後讓使用者失去追溯能力

## 10. 視覺方向

延續目前 `Design B`：

- 區塊之間有明確間距
- 區塊內排版緊密
- 左右欄位靠排版層級區分，不依賴厚重框線
- 分類名稱應比項目名稱更具層級
- 項目數量與狀態使用低對比輔助文字

## 11. 推薦落地順序

1. 新增整合式 `Master Data` 頁面
2. 先保留原 `分類管理` / `項目管理` 頁面作為過渡
3. 完成分類清單 + 右側項目區的基本互動
4. 補搜尋與篩選
5. 穩定後再決定是否移除舊頁

## 12. 結論

你的想法是對的：

- 分類與項目應整合
- 項目應該依分類群組呈現

若以可用性與後續維護性來看，最適合的操作不是單純把兩頁拼在一起，而是做成：

- 單一頁面
- 分類作為主要導航
- 項目作為當前分類下的明細內容

這會是最符合實際維護情境的版本。
